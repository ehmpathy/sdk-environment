# vision: cli binary for commit parsing

## the outcome world

### day-in-the-life

a developer adds sdk-environment to their service. at deploy time, instead of writing custom git commands to extract commit info, they call the sdk's binary:

```bash
export COMMIT=$(npx sdk-environment get commit)
npx sls deploy --stage $SLS_STAGE
```

the binary outputs the commit slug to stdout. done. no more bespoke parsing.

### before/after contrast

**before (svc-jobs today)**:
```bash
# line 58 of package.json
"deploy:release": "... && export COMMIT=\"v$npm_package_version@$(git rev-parse --short HEAD)\" && npm run build && sls deploy ..."
```

problems:
- manually composing format
- no dirty detection
- no tag vs branch logic
- diverges from sdk-environment's canonical format
- each consumer reinvents the wheel

**after**:
```bash
"deploy:release": "export COMMIT=$(npx sdk-environment get commit) && npm run build && sls deploy ..."
```

benefits:
- single source of truth for commit format
- dirty detection included
- tag precedence over branch included
- format: `$gitref@$hash` or `$gitref@$hash+`

### the "aha" moment

when the developer realizes they can delete their custom git parsing logic and just call the binary — and get dirty detection + tag precedence for free.

## user experience

### usecases

| usecase | goal |
|---------|------|
| serverless deploy | pass commit slug as env var to lambda |
| docker build | tag image with commit slug |
| ci artifact | name artifact with commit slug |
| observability | embed commit in logs/metrics/traces |

### contract inputs & outputs

**input**: none (reads from git in cwd)

**output**: commit slug to stdout
- format: `$gitref@$hash` (e.g., `v1.2.3@abc1234` or `main@abc1234`)
- with dirty suffix: `main@abc1234+`
- exit codes (per `rule.require.exit-code-semantics`):
  - 0 = success
  - 1 = malfunction (git failed unexpectedly)
  - 2 = constraint (not a git repo, no ref found)

### what it looks like to use

```bash
# basic usage
npx sdk-environment get commit
# output: v1.2.3@abc1234

# capture to env var
export COMMIT=$(npx sdk-environment get commit)

# use in serverless.yml
environment:
  COMMIT: ${env:COMMIT}

# use in dockerfile
ARG COMMIT
ENV COMMIT=$COMMIT
```

### timelines

1. **install**: already have sdk-environment as dep
2. **call**: one binary invocation at build/deploy time
3. **done**: commit slug captured, ready to pass to runtime

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| eliminate bespoke parsing | yes — one canonical source |
| dirty detection | yes — already in getEnvCommitFromGit |
| tag precedence | yes — already in getEnvCommitFromGit |
| easy adoption | yes — one binary call |

### pros

- single source of truth for commit format
- reuses extant `getEnvCommitFromGit` logic
- zero config — just call the binary
- already have async + sync variants

### cons

- adds `npx` overhead at deploy time (~1-2s)
- requires sdk-environment as a dep (already true for consumers)

### edgecases

| edgecase | pit of success |
|----------|----------------|
| not a git repo | exit 2 (constraint), clear error to stderr |
| no tags | falls back to branch name |
| detached HEAD | uses commit hash as ref |
| dirty worktree | appends `+` suffix |
| git command fails | exit 1 (malfunction), error to stderr |

## what is awkward?

### format mismatch

svc-jobs currently uses `v$npm_package_version@hash` but our format is `$gitref@hash`. if the repo has a tag like `v0.31.9`, they'll match. but if deploying from a branch, they'll get `main@hash` instead of `v0.31.9@hash`.

**tradeoff**: do we add a `--format` flag to support custom formats? or accept that canonical format is good enough?

### bun requirement

rhachet uses bun-compiled binaries for speed. this sdk follows the same pattern:
- bun required for the binary
- no fallback needed

most ci environments can install bun. the speed benefit (~1s vs ~2s startup) is worth it for frequent deploy-time calls.
