# vision: sync + async commit resolver

## the outcome world

### day-in-the-life with this in place

a developer who builds a logger needs commit info at module-load time for traceability. today they face a choice:

**before**
```ts
// option A: async at startup (awkward)
let environment: Environment;
const init = async () => {
  environment = await getEnvironment(); // has commit from git
};
init();

// option B: sync but no git commit info
const environment = getEnvironment.static(); // commit requires COMMIT envar
```

**after**
```ts
// sync AND git commit info — works by default!
const environment = getEnvironment.static();
// commit = 'main@abc123' from git, synchronously

// or with explicit parsers if customization needed
const environment = getEnvironment.static({
  parsers: {
    commit: [fromGit.sync(), fromEnvar('COMMIT')],
  },
});
```

### before/after contrast

| before | after |
|--------|-------|
| `fromGit()` is async-only | `fromGit.sync()` for sync contexts |
| `.static()` skips git → needs COMMIT envar | `.static()` includes `fromGit.sync()` by default |
| sync users lose commit traceability | sync users get full commit info out of the box |

### the "aha" moment

> "wait, i can get the git commit info in my logger without async?"

yes. `fromGit.sync()` runs `execSync` under the hood, returns immediately.

## user experience

### usecases

| usecase | goal | contract |
|---------|------|----------|
| sync logger init | embed commit in every log line | `fromGit.sync()` in `.static()` |
| async service bootstrap | full environment detection | `fromGit.async()` or `fromGit()` |
| backwards compat | keep extant code functional | `fromGit()` remains async |

### contract inputs & outputs

```ts
// sync variant
fromGit.sync(): () => EnvironmentCommitSlug | null

// async variant (explicit)
fromGit.async(): () => Promise<EnvironmentCommitSlug | null>

// default (async, backwards compat)
fromGit(): () => Promise<EnvironmentCommitSlug | null>
```

### what it looks like to use them

**sync usage with `.static()`**
```ts
import { getEnvironment, fromGit, fromEnvar } from 'sdk-environment';

// sync: use fromGit.sync()
const env = getEnvironment.static({
  parsers: {
    commit: [fromGit.sync(), fromEnvar('COMMIT')],
  },
});

console.log(env.commit); // 'feat/auth@abc123'
```

**async usage (unchanged)**
```ts
import { getEnvironment, fromGit } from 'sdk-environment';

// async: fromGit() works as before
const env = await getEnvironment({
  parsers: {
    commit: [fromGit()],
  },
});
```

**explicit async**
```ts
// explicit: fromGit.async() for clarity
const env = await getEnvironment({
  parsers: {
    commit: [fromGit.async()],
  },
});
```

### timeline

1. user imports `fromGit`
2. user calls `fromGit.sync()` or `fromGit.async()` to get parser
3. user passes parser to `getEnvironment.static()` or `getEnvironment()`
4. parser runs git commands (sync or async)
5. returns `'ref@hash'` or `'ref@hash+'` if dirty

## mental model

### how users describe this to a friend

> "fromGit now has .sync() and .async() so you can pick which one fits your context"

### analogies

| analogy | fits because | difference |
|---------|--------------|------------|
| `fs.readFile` vs `fs.readFileSync` | sync/async variants for same operation | node uses `*Sync` suffix, not `.sync()` method |
| `JSON.parse` vs `fetch().json()` | sync vs async json parse | n/a |

**note**: node.js convention is `*Sync` suffix (e.g., `readFileSync`), not `.sync()` method. the proposed `.sync()` / `.async()` is a variation — more discoverable via autocomplete but differs from stdlib pattern.

### terms they'd use vs terms we'd use

| user term | our term |
|-----------|----------|
| "sync version" | `fromGit.sync()` |
| "async version" | `fromGit.async()` |
| "the default" | `fromGit()` → async |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| sync git commit info | ✓ | `fromGit.sync()` |
| backwards compat | ✓ | `fromGit()` remains async |
| pit of success | ✓ | type system prevents sync/async mismatch |

### pros

- **no break** — `fromGit()` behavior unchanged
- **symmetric API** — `.sync()` / `.async()` mirrors node conventions
- **type safety** — `SyncParser` vs `AsyncParser` enforced by types
- **discoverable** — dot notation shows options in autocomplete

### cons

- **sync git blocks** — `execSync` blocks event loop; acceptable for module init
- **two implementations** — sync and async codepaths must stay in sync
- **name overlap** — `fromGit.async()` is verbose when `fromGit()` already is async

### edge cases and pit of success

| edge case | pit of success |
|-----------|----------------|
| user passes `fromGit.async()` to `.static()` | **type error** — `AsyncParser` incompatible with `SyncParser[]` |
| user passes `fromGit.sync()` to `.filled()` | **works** — sync parsers satisfy async interface |
| no git repo | both variants return `null` gracefully |
| dirty repo | both detect dirty via `git status --porcelain` |

## open questions & assumptions

### assumptions

1. **`execSync` is acceptable** — block git commands at module load is tolerable
2. **name convention** — `.sync()` / `.async()` aligns with user expectations
3. **backwards compat** — `fromGit()` must remain async

### answered questions

all questions triaged via logic and extant code analysis:

1. **[answered]** api design: which option?
   - option A: `fromGit()` → async, `fromGit.sync()` → sync ✓
   - option B: `fromGit.sync()`, `fromGit.async()`, no bare `fromGit()`
   - option C: `fromGit({ sync: true })` config object style
   - **answer**: option A. backwards compat is a hard constraint (semver). only option A preserves `fromGit()` behavior.

2. **[answered]** default parsers: should `static()` include `fromGit.sync()`?
   - **answer**: yes. update `DEFAULT_COMMIT_PARSERS_SYNC` to include `fromGit.sync()`.
   - wisher decision: convenience outweighs the 50-200ms cost for sync users who want commit info.
   - async defaults remain unchanged (`fromGit()` already included).

3. **[answered]** build-time alternative?
   - **answer**: this is not a question — the wisher chose runtime. build-time is an alternative that could be documented in "what is awkward?" but is not part of this wish.
   - moved to informational note below.

4. **[answered]** shallow clone support?
   - **answer**: extant code already handles this. see `getEnvCommitFromGit.ts:62-69`:
     - tag lookup fails gracefully → returns `null`
     - fallback to branch name via `git rev-parse --abbrev-ref HEAD`
     - shallow clones work: they use branch name instead of tag

none — no external dependencies referenced.

## what is awkward?

### name overlap

`fromGit()` is already async. `fromGit.async()` is redundant but provides symmetry.

options:
- keep both: `fromGit()` (async) + `fromGit.sync()` + `fromGit.async()` (explicit)
- drop bare: only `fromGit.sync()` + `fromGit.async()`

### sync blocks event loop

`execSync` blocks. for module-level init this is fine, but users might misuse in request handlers. mitigations:
- document the block nature
- rely on users to choose sync consciously

**usage guidance required**: docs must explicitly state:
- ✓ use at module top-level init
- ✓ use in cli startup
- ✗ do NOT use in request handlers
- ✗ do NOT use in hot paths

**docs must prominently warn**: "fromGit.sync() blocks for git exec (typical: 50-200ms, varies by repo size and disk speed). use only at module init."

### two codepaths

sync and async implementations must stay synchronized. changes to one must apply to both. mitigations:
- extract shared logic into helper functions
- test both variants with same test cases

### alternative: build-time injection

users could avoid runtime git exec entirely via build-time injection:

```bash
# in CI/CD
export COMMIT="$(git describe --tags --always)@$(git rev-parse --short HEAD)"
```

this avoids the sync block but requires CI/CD setup. the sync `fromGit.sync()` approach works without CI/CD changes, which is why the wisher chose runtime.
