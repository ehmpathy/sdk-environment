### blockers that require human review

these blockers appear to be design decision conflicts, not code defects:

#### 1. mutable shared state in cache (3 mentions across reviews)

**files**: getEnvironment.ts:85-90, 218-221, 263-266

**found**: cache object uses mutable state

**rationale for current design**:
- cache pattern is standard SDK behavior for expensive operations
- documented with `.note = intentional mutable state for cache`
- `cache: 'skip'` escape hatch provided for tests/special cases
- promise-based cache prevents race conditions (concurrent calls await same promise)

#### 2. race condition in promise cache assignment

**file**: getEnvironment.ts:194-206

**found**: concurrent calls could both assign promises

**rationale for current design**:
- this is a false positive — the code ALREADY caches the promise before computation
- comment says `// create and cache the promise before computation begins`
- deliberate mutation enables concurrent callers to await same promise

#### 3. optional input parameters

**files**: getEnvironment.ts:165-166, 230-231, 280-281

**found**: `input?: {...}` allows undefined

**rationale for current design**:
- public SDK API — optional params improve UX
- rule exception for `src/contract/` applies (public interface)
- callers should not need to pass `null` for default behavior

#### 4. runParsers for-loop decode friction

**files**: getEnvironment.ts:125, 153

**found**: for loops require mental simulation

**rationale for current design**:
- `runParsers` is a leaf operation (transformer), not an orchestrator
- rule applies to orchestrators only per `define.domain-operation-grains`
- code uses named `{ parser, name }` zipped objects, not positional indices
- pattern is standard "find first match" which is atomic behavior

#### 5. premature abstraction (nitpick)

**found**: zipParsersWithNames used fewer than 3 times

**counter-evidence**: grep shows 6 usages at lines 158, 163, 168, 232, 237, 242

## recommendation

signal stone as **blocked** for human review. all 7 blockers are design decisions that require human judgment, not mechanical code fixes:

1. cache pattern is intentional SDK behavior
2. optional params are correct for public API UX
3. runParsers is a leaf operation, rule scope is orchestrators
4. zipParsersWithNames has 6 usages (review error)

human should decide whether to:
- override blockers as acceptable design decisions
- refactor cache to different pattern (e.g., external WeakMap)
- change public API to require explicit null
