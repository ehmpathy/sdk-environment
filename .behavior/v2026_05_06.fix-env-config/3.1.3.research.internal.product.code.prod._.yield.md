## pattern 3: parser chain (first-wins precedence) [REUSE]

parsers run in order until first non-null result.

**citation [3]** `src/domain.operations/getEnvironment/getEnvironment.ts:91-112`
```ts
const runParsers = async <T>(
  parsers: AsyncParser<T>[],
  parserNames: string[],
  validate: (value: unknown) => value is T,
  attributeName: string,
): Promise<T> => {
  for (let i = 0; i < parsers.length; i++) {
    const parser = parsers[i]!;
    const result = await parser();
    if (result !== null) {
      if (!validate(result)) {
        throw new BadRequestError(
          `invalid ${attributeName} value: ${JSON.stringify(result)}. parser: ${parserNames[i]}`,
        );
      }
      return result;
    }
  }
  throw new UnexpectedCodePathError(
    `could not derive ${attributeName}. tried parsers: ${parserNames.join(', ')}`,
  );
};
```

**relation to wish:** config will use this same pattern. however, config parsers have additional constraint: they must validate against access (forced config for prod/test, flexible for prep).

## pattern 8: fromEnvar factory [REUSE]

generic factory creates typed envar parsers.

**citation [8]** `src/domain.operations/parsers/fromEnvar.ts:1-15`
```ts
/**
 * .what = factory to create envar parser for any environment variable
 * .why = enables fromEnvar('ACCESS'), fromEnvar('SERVER'), etc.
 */
export const fromEnvar = <T extends string>(
  envarName: string,
): (() => T | null) => {
  const parser = (): T | null => {
    const value = process.env[envarName];
    if (!value || value === '') return null;
    return value as T;
  };
  Object.defineProperty(parser, 'name', { value: `fromEnvar(${envarName})` });
  return parser;
};
```

**relation to wish:** we will use this factory for CONFIG envar.

## pattern 9: type guard validator [EXTEND]

validators use type guard pattern with allowlist.

**citation [9]** `src/domain.operations/validators/isEnvironmentAccessTier.ts:1-16`
```ts
import type { EnvironmentAccessTier } from '../../domain.objects/EnvironmentAccessTier';

const ACCESS_TIERS: readonly EnvironmentAccessTier[] = ['test', 'prep', 'prod'];

/**
 * .what = type guard for EnvironmentAccessTier
 * .why = validates parser output at runtime
 */
export const isEnvironmentAccessTier = (
  value: unknown,
): value is EnvironmentAccessTier => {
  return (
    typeof value === 'string' &&
    ACCESS_TIERS.includes(value as EnvironmentAccessTier)
  );
};
```

**relation to wish:** we will create `isEnvironmentConfigSlug` that validates the slug starts with a valid tier:
```ts
export const isEnvironmentConfigSlug = (
  value: unknown,
): value is EnvironmentConfigSlug => {
  return (
    typeof value === 'string' &&
    ACCESS_TIERS.some((tier) => value.startsWith(tier))
  );
};
```

## key constraint: access-config validation

unlike other attributes, config has a constraint: it must be compatible with access.

- prod access → config must start with 'prod' (forced)
- test access → config must start with 'test' (forced)
- prep access → config can be 'test', 'prep', or extended slug (flexible)

this validation happens after both access and config are parsed, as a post-parse check.
