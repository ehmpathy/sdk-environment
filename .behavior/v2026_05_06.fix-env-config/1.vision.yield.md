# vision: config separate from access

## the outcome world

### before: conflated concerns

```ts
const environment = await getEnvironment();

// access drives config implicitly
const config = await getConfig({ tier: environment.access });
// problem: prep access always loads prep config, even for tests
```

a developer authenticated to their org's prep AWS account runs `npm test`. their tests need test secrets (mock api keys, local db passwords). but `getEnvironment().access` returns `'prep'` because that's which account they're authenticated to. the config loader fetches prep secrets. tests fail or worse — accidentally modify prep data.

workaround: manually set `ACCESS=test` before tests run. but this lies — you don't have test access, you have prep access with test config intent.

### after: separated concerns

```ts
const environment = await getEnvironment();
// environment.access = 'prep' (which AWS resources)
// environment.config = 'test' (which secrets to load)

const config = await getConfig({ tier: environment.config });
// loads test secrets correctly
```

the developer runs `npm test`. NODE_ENV=test signals config intent. `access` stays `'prep'` (truth), `config` becomes `'test'` (intent). tests load test secrets. prep resources stay untouched.

### the "aha" moment

> "oh — access is what i *can* touch, config is what i *want* to touch"

the separation clicks when you realize most orgs run tests while authenticated to prep:
- they don't have a separate test AWS account
- test secrets live in prep account's secret manager
- NODE_ENV=test signals "load test config" not "switch to test account"

## user experience

### usecase: run tests against prep account

**goal:** run integration tests that need real aws resources but test-specific secrets

**contract:**
```ts
// input: NODE_ENV=test (set by test runner)
// input: authenticated to prep AWS account (via aws sso)

const env = await getEnvironment();
// output: { access: 'prep', config: 'test', server: '...', commit: '...' }
```

**timeline:**
1. developer runs `npm test`
2. jest sets NODE_ENV=test
3. test code calls `getEnvironment()`
4. access parser chain → 'prep' (from aws account)
5. config parser chain → 'test' (from NODE_ENV)
6. config loader uses `env.config` to fetch test secrets
7. tests run with test secrets but prep resources

### usecase: explicit config override

**goal:** run prep code locally with explicit prep config (not test defaults)

**contract:**
```ts
// input: CONFIG=prep (explicit override)
// input: NODE_ENV=test (jest runs)

const env = await getEnvironment();
// output: { access: 'prep', config: 'prep', ... }
// CONFIG envar takes precedence over NODE_ENV
```

### usecase: production is non-negotiable

**goal:** production access always means production config (safety)

**contract:**
```ts
// input: authenticated to prod AWS account

const env = await getEnvironment();
// output: { access: 'prod', config: 'prod', ... }
// no override allowed — prod access → prod config, always
```

## mental model

### how users describe it

> "access is which account i'm logged into. config is which secrets to load. usually they match, but for tests they don't."

### analogies

**hotel keycard:** your keycard (access) opens certain doors. the room service menu (config) determines what you order. a master keycard opens all doors but you still pick from different menus.

**bank account:** your login (access) determines which accounts you can see. the transaction filter (config) determines which ones you're work on.

### terms

| user says | we say | semantic |
|-----------|--------|----------|
| "which account am i in" | `access` | AWS account authentication (tier: test/prep/prod) |
| "which secrets to use" | `config` | secret/option selection (slug: extensible, e.g., `prod:v2023_08_21`) |
| "config" | `config` (via CONFIG envar) | our canonical term |
| "environment" | `Environment` (the full object) | the complete runtime context |

**tier vs slug:**
- `EnvironmentAccessTier` = fixed set: `'test' | 'prep' | 'prod'`
- `EnvironmentConfigSlug` = template literal: `` `${EnvironmentAccessTier}${string}` `` — must start with a tier, can extend (e.g., `prod`, `prod:v2023_08_21`)

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| run tests with test config while in prep account | config derived from NODE_ENV, separate from access |
| explicit config override | CONFIG envar takes precedence |
| prod safety | prod access forces prod config (no override) |
| backwards compatible | extant code that uses only `access` continues to work |

### pros

- **truthful access:** `access` reflects actual AWS authentication, never lies
- **intent-driven config:** `config` reflects what you want, respects NODE_ENV
- **prod safety:** prod access → prod config is mandatory
- **explicit override:** CONFIG envar for edge cases
- **backwards compatible:** `access` unchanged for extant consumers

### cons

- **new concept:** users must learn access vs config distinction
- **potential confusion:** "why is access different from config?"
- **migration:** extant code that uses `access` for config load needs update

### edgecases and pit of success

| edgecase | behavior | pit of success |
|----------|----------|----------------|
| prod access + CONFIG=test | error (failfast) | prod is non-negotiable, don't silently ignore |
| test access + CONFIG=prep | error (failfast) | test access forces test config, don't silently ignore |
| prep access + CONFIG=test | config = 'test' | explicit override honored |
| prep access + NODE_ENV=test | config = 'test' | test runner signal honored |
| prep access + CONFIG=prep | config = 'prep' | explicit override honored |
| prep access + NODE_ENV=test + CONFIG=prep | config = 'prep' | explicit CONFIG beats NODE_ENV |
| prep access + no signals | config = 'prep' | defaults to access |
| no access detected | error | fail fast, don't guess |

**config precedence for prep access:**
1. CONFIG envar (explicit override)
2. NODE_ENV=test (test runner signal)
3. defaults to 'prep'

**forced config (no override allowed):**
- prod access → prod config
- test access → test config

## open questions & assumptions

### assumptions

1. **most orgs don't have test accounts** — test and prep secrets live in same AWS account (per wish: "specifically, most orgs never have a test account. they typically collocate their test config within their prep accounts")
2. **NODE_ENV=test is reliable signal** — test runners (jest, vitest) set this
3. **CONFIG envar is standard** — industry practice for explicit override
4. **prod access must force prod config** — safety invariant

## what is awkward?

### the "config follows access by default" rule

if no config signals exist, config defaults to access. this feels natural but could confuse:
- "why did config change when i switched accounts?"
- answer: because you didn't set CONFIG explicitly

### prod override prohibition

we say CONFIG=test is ignored when access=prod. this might surprise users who expect explicit envars to win. but the safety benefit outweighs the surprise.

### NODE_ENV overload

NODE_ENV=test already means "tests run" for jest/vitest. now it also means "load test config". this is intentional (most test runners set it), but could confuse if someone sets NODE_ENV=test for other reasons.

**caveat:** not all test runners set NODE_ENV=test. jest and vitest do by default. mocha and ava do not — they leave it to the user. users of mocha/ava must set CONFIG=test or NODE_ENV=test explicitly.

### name: config vs tier

we use `config` for the concept. CONFIG envar is the canonical signal. internal terminology is `config`.
