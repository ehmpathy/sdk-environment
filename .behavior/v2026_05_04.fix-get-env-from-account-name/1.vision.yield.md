# vision: account name fallback for env access detection

## the outcome world

### before

```
getEnvironment()
  → access parser chain: fromEnvar → fromAwsAccountAlias → fromNodeEnv
  → alias not set on account
  → fallback to NODE_ENV
  → NODE_ENV not set in lambda
  → UnexpectedCodePathError: "could not derive access"
```

developer scratches head. lambda is clearly running in prod. but alias was never configured on this account.

### after

```
getEnvironment()
  → access parser chain: fromEnvar → fromAwsAccountAlias → fromAwsAccountName → fromNodeEnv
  → alias not set? no problem
  → account name is "myorg-prod"
  → matches '*-prod' pattern
  → { access: 'prod', server: 'lambda', commit: '...' }
```

just works. account name is always present (required at account creation).

### the "aha" moment

"wait, why do i even need to set an alias? my account is literally named `acme-prod`. the sdk should just figure it out."

now it does.

## user experience

### usecases

| goal | how |
|------|-----|
| detect prod/prep/test from aws | `await getEnvironment()` — alias-first, name-fallback |
| custom name patterns | `fromAwsAccountName({ map: { '*-production': 'prod' } })` |
| explicit parser chain | `getEnvironment({ parsers: { access: [fromAwsAccountName()] } })` |

### contract

```typescript
// new parser (mirrors alias parser exactly)
import { fromAwsAccountName } from 'sdk-environment';

// factory returns parser function
const parser = fromAwsAccountName();           // default: *-prod, *-prep, *-test
const parser = fromAwsAccountName({ map });    // custom patterns

// map interface (same as alias)
interface AwsAccountNameMap {
  [pattern: string]: EnvironmentAccessTier;    // 'prod' | 'prep' | 'test'
}

// example custom map
const map = {
  'acme-production': 'prod',     // exact match
  '*-preproduction': 'prep',     // glob
  '*-dev': 'test',
};
```

### timeline

1. **install**: `npm i sdk-environment` (no change — feature ships with sdk)
2. **use**: `await getEnvironment()` — name parser in default chain
3. **customize**: `fromAwsAccountName({ map })` — override patterns if needed

### namespace access

```typescript
import { getEnvAccess } from 'sdk-environment';

// direct invocation
const tier = await getEnvAccess.fromAwsAccountName();

// in custom chain
getEnvironment({
  parsers: {
    access: [
      getEnvAccess.fromEnvar,
      getEnvAccess.fromAwsAccountAlias,
      getEnvAccess.fromAwsAccountName,    // new
      getEnvAccess.fromNodeEnv,
    ],
  },
});
```

## mental model

### how users describe it

> "sdk-environment figures out if you're in prod or test by looking at your aws account — tries the alias first (if you set one), then falls back to the account name."

### analogies

- **dns fallback**: like how dns tries multiple resolvers — alias is primary, name is secondary
- **path resolution**: like how shells check PATH entries in order — first match wins

### terms

| user says | sdk says |
|-----------|----------|
| "account name" | `fromAwsAccountName` |
| "account alias" | `fromAwsAccountAlias` |
| "environment" | `access` (tier: prod/prep/test) |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| accounts without alias configured | yes — name always present |
| no new required permissions | yes — `account:GetAccountInformation` is read-only on own account |
| no organizations required | yes — works on standalone accounts |
| backwards compatible | yes — alias still takes precedence |
| enabled by default | yes — in default parser chain; users can opt-out via custom chain |

### pros

- **always available**: account name is required at creation
- **zero config**: works with default `*-prod/*-prep/*-test` patterns
- **composable**: same factory pattern as other parsers
- **explicit precedence**: alias-first maintains intentionality

### cons

- **accidental match risk**: name could match pattern unintentionally (e.g., "my-prod-sandbox")
- **less intentional**: alias is explicitly set for this purpose; name might just happen to match
- **extra api call**: adds latency when alias returns null

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| name matches by accident | alias-first order → intentional alias always wins |
| name doesn't match any pattern | parser returns null → falls through to next |
| no aws credentials | parser catches auth errors → returns null → falls through |
| rate limited | parser catches throttle → returns null → falls through |
| account in pending/suspended state | api still returns name → works |

## what is awkward?

### map type naming

`AwsAccountAliasMap` works for alias but feels wrong for name. options:
1. duplicate type as `AwsAccountNameMap` (redundant)
2. rename to `AwsAccountPatternMap` (breaking change for alias users importing type)
3. keep `AwsAccountAliasMap` and document it works for both (confusing)

**recommendation**: introduce `AwsAccountPatternMap` as new canonical name, re-export `AwsAccountAliasMap` as alias for backwards compat.

### accidental match risk

the wish acknowledges this: "is accidental name match a real risk?"

mitigations:
- alias-first order (intentional alias wins)
- users control their own naming conventions within their org
- custom map allows explicit patterns

this feels acceptable. the value of "just works" outweighs the edge case of accidental match.

### two aws api calls

when alias returns null, we make a second api call for name. adds latency.

mitigations:
- both calls are fast (<100ms typically)
- results are cached by sdk-environment
- users can customize parser chain to skip alias if they never use it

acceptable tradeoff for broader compatibility.
