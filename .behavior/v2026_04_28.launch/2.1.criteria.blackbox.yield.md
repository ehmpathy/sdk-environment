# blackbox criteria: sdk-environment

## usecase.1 = get environment with defaults

given a runtime with detectable environment signals
  when getEnvironment.filled() is called with no arguments
    then it returns an Environment object
      sothat users have typed access to environment info
    then the access property is 'test' | 'prep' | 'prod'
      sothat users can route behavior by access level
    then the server property matches ServerTier format
      sothat users can identify where code runs
    then the commit property contains git ref and hash
      sothat users can trace logs to exact code

given a runtime with no detectable environment signals
  when getEnvironment.filled() is called with no arguments
    then it throws an error with clear message
      sothat users know which attribute failed to derive
    then the error includes which parsers were tried
      sothat users can debug parser configuration

---

## usecase.2 = get environment with custom parsers

given a user with custom infrastructure
  when getEnvironment.filled({ access: [...] }) is called
    then the custom parsers run in provided order
      sothat users control parser precedence
    then the first parser to return non-null wins
      sothat behavior is predictable
    then remaining parsers are skipped
      sothat performance is optimal

given a user mixing custom and default parsers
  when getEnvironment.filled({ access: [...] }) is called
    then only the specified attribute uses custom parsers
      sothat users can override selectively
    then other attributes use default parsers
      sothat users don't need to configure everything

---

## usecase.3 = sync vs async environment access

given a runtime where async parsers have not yet run
  when getEnvironment.static() is called
    then it returns Environment from sync parsers only
      sothat users can access environment without await
    then async parsers are skipped
      sothat the call is truly synchronous

given a runtime where filled() has been called
  when getEnvironment.static() is called
    then it returns its own cached result (not filled's cache)
      sothat static and filled are independent paths

given a runtime where no sync parsers can derive a value
  when getEnvironment.static() is called
    then it throws an error
      sothat users know sync path cannot derive environment

---

## usecase.4 = cache behavior

given a runtime where filled() has been called once
  when getEnvironment.filled() is called again
    then it returns the cached result
      sothat repeated calls are fast
    then parsers do not run again
      sothat expensive operations are not repeated

given a user who needs fresh environment data
  when getEnvironment.filled({ cache: 'skip' }) is called
    then parsers run fresh
      sothat users can force re-evaluation
    then the cache is updated with new result
      sothat subsequent calls use fresh data

given a runtime where static() has been called once
  when getEnvironment.static() is called again
    then it returns the cached result
      sothat repeated sync calls are fast

given a user who needs fresh static data
  when getEnvironment.static({ cache: 'skip' }) is called
    then sync parsers run fresh
      sothat users can force re-evaluation on sync path

---

## usecase.5 = access value validation

given a parser that returns 'test'
  when getEnvironment.filled() is called
    then access is 'test'
      sothat test environments are identified

given a parser that returns 'prep'
  when getEnvironment.filled() is called
    then access is 'prep'
      sothat pre-production environments are identified

given a parser that returns 'prod'
  when getEnvironment.filled() is called
    then access is 'prod'
      sothat production environments are identified

given a parser that returns an invalid access value like 'development'
  when getEnvironment.filled() is called
    then it throws a validation error
      sothat invalid values fail fast
    then the error message includes the invalid value
      sothat users can fix their parser

## usecase.7 = commit traceability

given a git repository with commits
  when getEnvironment.filled() is called
    then commit contains the git ref (tag or branch)
      sothat users know which ref is deployed
    then commit contains the short hash
      sothat users can checkout exact code

given a git repository with uncommitted changes
  when getEnvironment.filled() is called
    then commit ends with '+'
      sothat dirty deploys are visible in logs

given no git repository (e.g., docker without .git)
  when getEnvironment.filled() is called
    then commit falls back to envar (e.g., GIT_COMMIT)
      sothat containerized deploys still have traceability
