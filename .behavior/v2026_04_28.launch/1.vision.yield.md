# vision: sdk-environment

## the outcome world

### day-in-the-life with sdk-environment

**morning: developer starts coding**

sarah opens her laptop, runs `npm test`. her tests connect to an in-memory database, mock external apis, and run in seconds. she never thinks about configuration — `getEnvironment()` returns `{ access: 'test', server: 'local@unix', commit: 'feat/auth@a1b2c3d' }` and her app just works.

**afternoon: pr merged, ci runs**

her branch passes ci. the same code runs on github actions, but now `getEnvironment()` returns `{ access: 'test', server: 'local@cicd', commit: 'feat/auth@e4f5g6h' }`. logs include the commit hash automatically — if a flaky test fails, she can trace it to the exact code.

**evening: deploy to prep**

the lambda deploys to prep. `getEnvironment()` returns `{ access: 'prep', server: 'cloud@aws.lambda', commit: 'v1.2.3@i7j8k9l' }`. the app connects to the sandbox stripe api and prep database. no code changes, no env files — the parsers inferred everything from aws account alias and lambda runtime.

**incident: production debug**

a week later, an oncall engineer sees an error in datadog. the log includes `commit: 'v1.2.3@i7j8k9l'`. they checkout that exact commit, run locally against prod (with proper auth), and reproduce the bug in minutes.

### before/after contrast

| before | after |
|--------|-------|
| `if (process.env.NODE_ENV === 'production')` | `if (environment.access === 'prod')` |
| `process.env.STAGE \|\| process.env.ENV \|\| 'dev'` | `environment.access` |
| logs say "production" but which lambda? | logs say `cloud@aws.lambda` with commit hash |
| "works on my machine" | exact commit traced in logs |
| env var soup: `NODE_ENV`, `STAGE`, `ENVIRONMENT`, `DEPLOY_ENV` | one shape: `{ access, server, commit }` |

### the "aha" moment

> "wait, i can see the exact git commit in every log line? and it tells me if someone deployed with uncommitted changes?"

the commit traceability is the killer feature. when something breaks at 3am, `commit: 'main@abc123+'` tells you:
1. exactly what code is running
2. that someone deployed with uncommitted changes (the `+`)
3. how to reproduce locally

## user experience

### usecases

| usecase | goal | how sdk-environment helps |
|---------|------|---------------------------|
| test isolation | tests don't touch prod data | `access: 'test'` routes to test fixtures |
| environment-aware logging | structured logs in cloud, pretty in local | `server.split('@')[0]` gives tier |
| config loading | load secrets from right source | `access` determines ssm path |
| incident response | trace error to exact code | `commit` in every log |
| drift detection | catch dirty deploys | `commit` ends with `+` if dirty |

### contract inputs & outputs

**input:** optional parser configuration

```ts
// minimal — use defaults
const env = await getEnvironment();

// custom — add your parsers
const env = await getEnvironment({
  access: [getEnvAccess.fromEnvar, fromK8sNamespace, getEnvAccess.fromNodeEnv],
});
```

**output:** strongly-typed environment object

```ts
type ServerTier = `local@${string}` | `cloud@${string}`;

interface Environment {
  access: 'test' | 'prep' | 'prod';
  server: ServerTier;
  commit: string;
}
```

### timeline

1. **install** (`npm install sdk-environment`) — 10 seconds
2. **integrate** (add `getEnvironment()` call) — 2 minutes
3. **customize** (add custom parsers if needed) — 5-15 minutes
4. **forget** (it just works) — forever

> "it figures out where your code is running and gives you a consistent shape"

> "like how express figures out `req.ip` automatically — this figures out your environment automatically"

> "three questions: what can i touch, where am i, what code is this"

| analogy | what it illuminates |
|---------|---------------------|
| passport | identifies who you are; environment identifies where you run |
| gps | tells you location; server tells you runtime location |

### user terms vs our terms

| user might say | we say | why |
|----------------|--------|-----|
| "env" or "stage" | `access` | env is overloaded; access is what it controls |
| "dev" | `prep` | dev describes who, prep describes what |
| "platform" or "runtime" | `server` | server is where code executes |
| "version" or "build" | `commit` | commit is precise; version could mean semver |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| environment detection | excellent | pluggable parsers handle any infra |
| traceability | excellent | commit hash + dirty flag is unique |
| simplicity | good | three attributes cover most cases |
| extensibility | excellent | custom parsers are trivial |
| type safety | excellent | literal types for access, string for server/commit |

### pros

- **minimal api** — one function, three attributes
- **pluggable** — add parsers for any infrastructure
- **traceable** — commit hash in every log
- **type-safe** — `access` is a union literal, not `string`
- **found or fail** — no silent defaults; parsers find values or throw (pit of success)
- **independent axes** — access and server are orthogonal

### cons

- **async** — `getEnvironment.filled()` is async (git/aws calls)
- **first call cost** — parsers may be slow on first invocation

### edgecases and pit-of-success

| edgecase | how we handle |
|----------|---------------|
| no git repo | `commit` git parser returns null, envar parser may provide value |
| aws call fails | parser returns null, next parser tries |
| all parsers return null | throw clear error with which attribute failed |
| multiple parsers match | first wins (documented, predictable) |
| dirty working directory | `commit` ends with `+` (visible, traceable) |

### questions answered [answered]

1. **cache strategy** — leave to user per ehmpathy pattern (sdk-config takes user-supplied cache)
2. **default parsers** — the readme's parsers are correct; the readme is the spec
3. **git commands** — verified: `git describe --tags --always`, `git rev-parse --short HEAD`, `git status --porcelain`
4. **access type** — use strict union ('test' | 'prep' | 'prod') per readme; sdk-config's looser string type is backwards compatible via typescript structural typing
5. **parser exception safety** — parsers are responsible for their own exception safety; they return T | null, never throw; getEnvironment trusts this contract

8. **sync variant** — `getEnvironment.static()` skips async parsers, returns sync; `getEnvironment.filled()` runs all parsers, returns async; default `getEnvironment()` is `filled` (wisher decision)
9. **error behavior** — throw (failfast) if all parsers return null for an attribute (wisher confirmed)
10. **server type** — use template literal `\`local@${string}\` | \`cloud@${string}\`` with runtime validation; no bare values (wisher decision)
11. **cache** — both `static()` and `filled()` cache independently with `with-simple-cache`; `{ cache: 'skip' }` forces refresh on either path (wisher decision)

## groundwork

### external research

**aws account alias:**
- use `@aws-sdk/client-iam` for minimal footprint (~50kb vs ~500kb)
- `ListAccountAliasesCommand` to get alias
- credentials from aws default credential chain
- parser returns null if no creds or error

**git commands:**
- `git describe --tags --always` — gives tag or short hash
- `git rev-parse --short HEAD` — gives short hash
- `git status --porcelain` — empty = clean, non-empty = dirty
- if not in git repo: parser returns null, falls back to envar

**lambda detection:**
- `LAMBDA_TASK_ROOT` envar is present in lambda
- `AWS_EXECUTION_ENV` contains runtime info
- these are reliable indicators

**ehmpathy patterns to follow:**

## what is awkward?

### `server` uses template literal types

the readme shows `server: string` but we use template literals for pit-of-success:

```ts
type ServerTier = `local@${string}` | `cloud@${string}`;
```

- autocomplete guides users to valid format (`local@`, `cloud@`)
- tier is type-enforced and parseable
- platform suffix remains open-ended
- runtime validation throws if invalid format

**resolved tradeoff:** structured flexibility via template literals (wisher decision).

### sync skips async parsers

`getEnvironment.static()` and `getEnvironment.filled()` run the same parser chain:

- `static()` → skips async parsers, returns sync result, caches independently
- `filled()` → runs all parsers (sync + async), caches independently
- `{ cache: 'skip' }` → forces fresh parser run on either path

**resolved tradeoff:** sync available by skip async parsers (wisher decision).

### first-wins vs merge

parsers are first-wins, not merge. if `fromEnvar('ACCESS')` returns `'prod'`, we don't run `fromAwsAccountAlias()`. this is:
- predictable
- fast (short-circuit)
- but means you can't "layer" parsers

**uncomfortable tradeoff:** can't combine signals from multiple sources.

### built-in cache for both paths

both `static()` and `filled()` cache independently with `with-simple-cache`:
- environment is immutable at runtime
- `{ cache: 'skip' }` forces fresh run on either path

**resolved tradeoff:** cache built-in for both paths (wisher decision).

### `commit` format is opinionated

`$gitref@$hash+` is our format. other tools use different formats:
- docker: `sha256:abc123`
- semver: `1.2.3`
- sentry: `abc123def456`

**uncomfortable tradeoff:** may not match extant observability tools' expectations.
